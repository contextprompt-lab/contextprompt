How repo scanning works

A four-stage pipeline that builds your codebase map before any meeting runs, then cross-references it against the transcript in real time.

1

File tree walk — gitignore-aware

contextprompt walks your repository from the root, respecting every .gitignore rule it finds. It automatically skips node_modules/, dist/, build/, and any other directories your project excludes. Only the files a developer would actually modify are indexed.

This prevents noise: no build artifacts, no vendored dependencies, no generated code polluting the file map that gets matched against your meeting transcript.

// Automatically skipped node_modules/ ← .gitignore rule dist/ ← .gitignore rule .git/ ← always excluded // Indexed for matching src/middleware/auth.ts src/routes/bots.ts src/server/db.ts
2

AST parsing — exports, types, function names

For TypeScript and JavaScript files, contextprompt uses the TypeScript compiler's AST parser (parse-only mode — no type checking) to extract exported identifiers: function names, class names, type aliases, and interface names.

This gives the extraction pipeline the vocabulary of your codebase. When someone says "update the AuthContext" in a meeting, the scanner already knows which file exports AuthContext — it doesn't need to guess.

// src/middleware/auth.ts — extracted exports export function requireAuth(req, res, next) export function requirePro(req, res, next) export type JWTPayload = { userId: string ... } export const JWT_SECRET
3

Token budgeting — 30k tokens per repo

Repository maps are capped at ~30,000 tokens to keep Claude API analysis fast and cost-efficient. When a repo exceeds this budget, progressive trimming applies: first dropping function signatures, then dropping deeply-nested files — preserving the highest-signal paths.

The result: your most important files (the ones closest to your project root, with the most exports) are always included. Deeply-buried utility files are dropped only when necessary.

4

Keyword matching — bigrams, trigrams, scoring

When a task is extracted from a meeting transcript, contextprompt scores every file in your repo map using bigram and trigram matching against the task description. File paths, function names, type names, and directory names all contribute to the score.

High-scoring files appear as the task's affected files. The extraction prompt then sends those files' full signatures to Claude for the final implementation steps — giving it the real context it needs to produce accurate, specific output.

// Transcript: "update the auth middleware to use JWT" // Keyword matches: src/middleware/auth.ts 0.94 ← "auth", "middleware" src/types/session.ts 0.71 ← "session" → JWT replacement src/routes/auth.ts 0.68 ← "auth" in path src/server/index.ts 0.12 ← low score, excluded

Key properties of the scanner

Designed for real-world codebases — not toy examples.

Parse-only, no compilation

contextprompt uses ts.createSourceFile() for AST parsing — not ts.createProgram(). There's no type checking, no import resolution, and no dependency on your node_modules. It works on any TypeScript or JavaScript project instantly, without a build step.

Respects .gitignore at every level

The walker processes .gitignore files at every directory level — not just the root. Monorepos with per-package ignore rules work correctly. Nested node_modules directories inside packages are automatically excluded.

Works with browser File System API

You can share your local repository via the browser's File System Access API — your source files never leave your machine. Only the file map (paths, function names, type signatures) is sent for analysis. For public repos, a GitHub URL also works.

Multiple repos per meeting

Connect a frontend repo and a backend repo simultaneously. contextprompt handles task attribution across both codebases — so "update the API endpoint and the React component that calls it" produces tasks in the right repo with the right file paths.

Multi-language aware

AST parsing applies to TypeScript and JavaScript files. For other languages (Python, Go, Java), contextprompt falls back to file path and directory name matching — still producing accurate results for common project structures.

Progressive trimming under budget

When a large repo hits the 30k token limit, the trimming algorithm preserves high-signal files: it drops function signatures first, then drops deeply-nested files. Root-level files with many exports are always kept, ensuring the most important context survives.

Technical questions about repo scanning

What developers ask about how the scanner works.

Does contextprompt upload my source code?

No. When you use the File System Access API (the browser-based path), contextprompt builds a repo map — file paths, function names, exported type signatures — and sends that map for analysis. Your actual source file contents stay in the browser. If you connect a public GitHub repository by URL, the public file content is fetched directly.

Does it work on private repositories?

Yes, in two ways. The File System Access API lets you share any local directory — public or private — without uploading files. Alternatively, if you use the CLI mode (for local audio capture), contextprompt scans your repos directly from disk, fully offline.

How accurate is the file path matching?

For codebases where function names and type names appear in transcript discussions, accuracy is very high — the bigram/trigram match of exported identifiers against the transcript is specific enough to identify the right file in most cases. Confidence levels in each extracted task reflect the match quality: high-confidence tasks have strong file matches; lower-confidence tasks may have fewer direct references in the transcript.

What languages does it support?

contextprompt provides full AST-level extraction (exported functions, types, interfaces) for TypeScript and JavaScript. For Python, Go, Java, Ruby, and other languages, it uses file path and directory structure matching — which is still highly effective for discussions that reference component names or module names that appear in file paths.

How large can my repository be?

There's no hard limit on repo size — the scanner is gitignore-aware and skips non-source files, so most production codebases fit well under the 30k token analysis budget. Very large monorepos may trigger progressive trimming (signatures dropped before deep files), but the most important files always remain in the map.
